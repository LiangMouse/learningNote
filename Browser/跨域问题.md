# 跨域问题与解决方案

## 是什么

跨域问题是指浏览器基于**同源策略**对不同源的请求进行限制时所产生的问题。同源策略是浏览器的一种安全机制，它要求从同一个源（协议、域名、端口）发起的请求才能共享资源。如果请求的资源来自不同的源，浏览器会限制这些请求，导致跨域问题。

## 解决方法

1. `JSONP`原生 Html 标签绕过跨域获得资源
2. 后端响应头的`CORS`，“官方”方式打开跨域的锁（最常用）
3. `Nginx`等的请求，本质上是将浏览器与服务器的请求转到服务器与服务器的请求，不受浏览器跨域控制

### JSONP

由于 `<script>`标签的 src 属性可以请求其他域的数据，并且不会受到同源策略的限制，JSONP 利用了这一点。它的基本思想是动态创建一个 `<script> `标签，通过 `src` 属性请求一个跨域的 URL。服务端返回一段 `JavaScript` 代码，其中包含一个回调函数调用，将请求的数据作为参数传递给该回调函数。这样浏览器执行返回的脚本时，就能直接获得数据。

**支持跨域请求的标签**

以下 HTML 标签支持跨域网络请求,同时需要防范其带来的`CSRF`网络攻击

- img
- script
- link
- iframe
- audio 和 video
- source

**缺点**：不适合 React,Vue 等 MVVM 框架使用(全局回调函数污染)，仅支持`get`请求，容易受到`XSS`网络攻击，可读性和安全性差，无法通过状态码判断请求结果

### CORS

浏览器是否启用跨域保护机制是由后端的响应决定的，`CORS`（Cross-Origin Resource Sharing，跨域资源共享） 是一系列人为定义的、基于 HTTP 头跨域的响应头

而服务端自然可以通过修改跨域响应头来实现对跨域场景的处理

`Access-Control-Allow-Origin`
`*`:对所有网址支持跨域
一些域名：对这些域名支持跨域操作
通常会和`OPTIONS`请求方法同时出现，`OPTIONS` 会起到一个预检（Preflight）的作用，询问目标服务器是否允许具体的跨域请求。

#### 简单请求（Simple Request）

满足以下全部条件的跨域请求，浏览器不会在正式请求前发送预检：

- 方法仅限：`GET`、`HEAD`、`POST`；
- 自定义请求头受限，仅允许：`Accept`、`Accept-Language`、`Content-Language`、`Content-Type`；
- 且 `Content-Type` 仅可为：`application/x-www-form-urlencoded`、`multipart/form-data`、`text/plain`；

表现：浏览器直接发起请求；若服务端返回 `Access-Control-Allow-Origin`（以及需要时的 `Vary: Origin`），且未涉及凭证（cookie）或已正确配置 `Access-Control-Allow-Credentials`，响应可被前端读取。

常见场景：普通表单提交、简单的 `fetch('/api', { method: 'POST', headers: { 'Content-Type': 'application/x-www-form-urlencoded' } })`。

#### 预检请求（Preflight Request）

不满足简单请求条件时，浏览器会在正式请求前先发送 `OPTIONS` 预检，请求头常见包含：

- `Origin`：请求来源；
- `Access-Control-Request-Method`：后续实际请求的方法；
- `Access-Control-Request-Headers`：后续实际请求将携带的自定义请求头列表。

服务端需在预检响应中返回：

- `Access-Control-Allow-Origin`：允许的来源；
- `Access-Control-Allow-Methods`：允许的方法；
- `Access-Control-Allow-Headers`：允许的自定义请求头；
- 可选 `Access-Control-Max-Age`：预检结果缓存秒数，减少重复预检。

表现：预检通过后，浏览器才会发起实际请求；若任一字段不匹配或缺失，浏览器阻止后续请求或屏蔽响应给前端脚本。

常见场景：

- 使用 `PUT`/`DELETE`/自定义方法；
- `POST` 但 `Content-Type: application/json`；
- 携带自定义头（如 `X-Token`、`Authorization` 等）。

### 反向代理

因为跨域实际上是由浏览器限制的，部署一个同域的服务器把请求代理到其他服务器就可以解决跨域问题，如 Nginx 用于生产环境，配置 Vite 用于开发环境

```JavaScript
// vite.config.js
import { defineConfig } from 'vite';
import vue from '@vitejs/plugin-vue';

export default defineConfig({
  plugins: [vue()],
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:5000',
        // any web you want
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
    },
  },
});
```

webpack 的 devServer

```javascript
//webpack.config.js
const path = require("path");

module.exports = {
  entry: "./src/index.js",
  output: {
    filename: "bundle.js",
    path: path.resolve(__dirname, "dist"),
  },
  devServer: {
    static: {
      directory: path.join(__dirname, "dist"),
    },
    port: 8080, // 本地开发服务器端口
    proxy: {
      // 代理跨域请求
      "/api": {
        target: "http://localhost:5000", // 后端服务器地址
        changeOrigin: true, // 修改请求头中的 Origin
        pathRewrite: { "^/api": "" }, // 可选：将 '/api' 替换为空字符串
      },
    },
  },
  mode: "development", // 开发模式
};
```
