# CSR, SSR 与 SSG

## 概念

`CSR`(客户端渲染)、`SSR`(服务端渲染)与 `SSG`(静态站点生成)是 Web 应用呈现内容的三种主流模式。

- CSR 是用户直接拿到残缺的 HTML 并由 JS 动态渲染；
- SSR 是在服务端直接返回完整的 HTML；
- SSG 则是在**构建阶段**提前生成好所有页面的静态 HTML 文件，用户访问时直接获取现成的文件，无需服务端实时计算。

## 常见问题

大致从三者区别，合适场景来进行对比。

### 三者的区别

CSR（Client-Side Rendering）：

- 浏览器下载 空 HTML，由 JavaScript 动态生成 DOM 并渲染。
- 数据通过 API 异步获取（如 fetch、axios）。
- 典型框架：React、Vue（默认模式）。
- 优点：交互性强，适合 SPA（单页应用），前后端分离开发方便。
- 缺点：首屏加载慢（需等 JS 执行完），SEO 不友好（爬虫难以抓取动态内容）。

SSR（Server-Side Rendering）：

- 服务器生成 完整 HTML 并返回给浏览器，浏览器直接渲染。
- 数据在服务器端获取并注入 HTML（每次请求发生时）。
- 典型框架：Next.js（React）、Nuxt.js（Vue）。
- 优点：首屏加载快，SEO 友好，数据实时性高。
- 缺点：服务器压力大（每次请求都要计算），交互性稍弱（需结合客户端 水合 hydration）。

SSG（Static Site Generation）：

- 构建系统生成 完整 HTML，部署在 CDN 或静态服务器上。
- 数据在 **构建时（Build Time）** 获取并固化进 HTML。
- 典型框架：Gatsby、Next.js/Nuxt.js（静态导出模式）。
- 优点：访问速度最快（纯静态文件读取），服务器成本极低，SEO 友好，安全性高。
- 缺点：数据时效性差（内容更新需重新构建部署），不适合内容频繁变动的大型站点，页面过多时构建时间长。

### 选择场景

由于 SSR 的首屏极致性能优化+SEO 友好的特点，使得 SSR 特别适合做需要实时数据的营销推广页或新闻门户。

而 CSR 由于现在也有很多性能优化解决方案+热更新开发的卓越体验，开发便利使得具体功能型应用页（如后台管理、SaaS 工具）倾向使用 CSR。

SSG 则由于其无需后端计算、响应速度极快的特性，特别适合内容相对固定、更新频率不高的场景，如技术文档、个人博客或企业官网。
