# Git 中 rebase 和 merge 的区别

1. **基本概念**：

   - `git merge`：将一个分支的变更合并到当前分支，通常会创建一个新的合并提交（merge commit）。它保持了两条历史线的独立性。
   - `git rebase`：将当前分支的变更"重放"到目标分支的最新提交上，改变历史提交的顺序。`rebase`不会产生合并提交，而是将当前分支的提交应用在目标分支之后。

2. **历史记录**：

   - `git merge` 会保留分支的合并历史，保留两条分支的历史，合并后会有一个新的提交记录来表示分支的合并。
   - `git rebase` 会重写当前分支的提交历史，使提交看起来像是直接从目标分支派生的。这通常使得历史看起来更简洁、更线性。

3. **使用场景**：

   - 使用 `git merge` 时，适合需要保留分支历史和合并记录的场景。
   - 使用 `git rebase` 时，适合希望使历史更简洁，避免过多的合并提交，特别是在清理提交历史时。

4. **冲突处理**：

   - 在 `git merge` 中，冲突会在合并时出现，需要手动解决冲突并提交。
   - 在 `git rebase` 中，冲突会在“重放”每个提交时逐个出现，需要逐个解决冲突并继续 rebase。

5. **优缺点**：
   - `git merge` 保持了历史的完整性，但可能会导致历史中出现多余的合并提交。
   - `git rebase` 使得历史更线性，更易于理解，但会改变历史，如果已经推送到远程仓库，需要小心使用，以避免破坏公共历史。

简单的总结：

- `merge`：保留历史，生成新的合并提交。
- `rebase`：重写历史，提交看起来更线性。
